# 新服务器与新域名部署完整流程

## 一、文档目的

本文基于当前仓库中的部署脚本、Docker Compose、Nginx 配置、前后端 Dockerfile 和 CI 配置，总结一套在以下场景下可直接执行的上线流程：

- 更换新的云服务器 IP
- 更换新的前台域名、后台域名
- 更换新的 SSL 证书
- 从本地构建镜像并推送到腾讯云镜像仓库
- 在云服务器上使用 docker compose 和部署脚本完成发布
- 程序改动后，删除旧镜像/旧容器并重新启动
- 域名 DNS 解析到新服务器
- 云服务器安全组与系统环境准备

本文按实际操作顺序编排，建议严格按顺序执行。

## 二、先看清当前仓库里的部署现状

当前仓库里实际上存在两套镜像发布思路：

- 生产部署脚本 deploy/deploy.sh 和 deploy/docker-compose.yml 默认从 GHCR 拉镜像，镜像地址格式是 ghcr.io/${GITHUB_REPO}/admin:${IMAGE_TAG} 与 ghcr.io/${GITHUB_REPO}/storefront:${IMAGE_TAG}
- 测试部署配置 deploy/docker-compose.test.yml 和 docs/docker-build-push-commands.md 使用的是腾讯云 TCR，镜像地址格式是 hkccr.ccs.tencentyun.com/website-pro/medusa_plus_pro:${IMAGE_TAG} 与 hkccr.ccs.tencentyun.com/website-pro/medusa_plus_front_pro:${IMAGE_TAG}

如果你这次要走“本地构建 -> 推送腾讯云 TCR -> 服务器脚本发布”的路线，那么生产环境必须先统一成腾讯云镜像源，否则 deploy.sh 启动时仍会去拉 GHCR 镜像。

也就是说，你有两个选择：

- 方案 A：继续使用 GHCR，不改 deploy/docker-compose.yml，只改 GITHUB_REPO 和服务器登录 GHCR 的方式
- 方案 B：统一改成腾讯云 TCR，这更符合你这次的目标

本文后续以方案 B 为主说明，同时指出需要改哪些文件。

## 三、上线前需要准备的固定信息

在开始之前，先把下面这些信息整理出来，后面会逐项填入配置：

- 新服务器公网 IP
- SSH 登录用户名
- 前台正式域名，例如 www.example.com
- 前台裸域名，例如 example.com
- 后台正式域名，例如 admin.example.com
- 前台域名证书文件
- 后台域名证书文件
- 腾讯云 TCR 仓库地址
- 腾讯云 TCR 用户名
- 腾讯云 TCR 密码
- PostgreSQL 用户名、密码、数据库名
- Redis 密码
- JWT_SECRET
- COOKIE_SECRET
- REVALIDATE_SECRET
- Medusa Publishable API Key

建议额外准备一个部署记录表，记录：

- 镜像 tag
- 部署时间
- 发布人
- 对应 git commit
- 服务器路径
- 当前证书到期时间

## 四、必须修改的配置文件清单

### 4.1 服务器部署目录中的配置

以下文件是这次换新服务器、新域名时一定要改的：

- deploy/.env
- deploy/docker-compose.yml
- deploy/nginx/conf.d/admin.conf
- deploy/nginx/conf.d/storefront.conf
- deploy/nginx/conf.d/00-block-ip-access.conf

### 4.2 代码仓库中可能需要改的配置

以下文件是否要改，取决于你是否改了域名、图片地址白名单或 CI 变量：

- my-store-storefront/next.config.js
- .github/workflows/deploy-storefront.yml
- .github/workflows/deploy-admin.yml

### 4.3 新域名下必须检查的代码点

#### 1. 后端上传文件 URL 前缀

后端 Medusa 生成图片地址时，使用的是 BACKEND_URL，对应逻辑在 my-store/medusa-config.ts。

必须保证：

- BACKEND_URL 指向后台公网地址加 /uploads
- 例如 https://admin.example.com/uploads

否则后台上传后的图片链接仍会指向旧域名。

#### 2. 前端图片白名单

my-store-storefront/next.config.js 里配置了 Next.js 的 remotePatterns。目前写死了：

- admin.wolzq.com
- www.wolzq.com
- wolzq.com

如果你更换成全新的域名，并且前端页面会通过 next/image 加载后台上传图片，那么这里必须改成新的域名，否则图片可能不显示。

#### 3. 前端构建时注入的 NEXT_PUBLIC 变量

my-store-storefront/Dockerfile 在 build 阶段注入：

- NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY
- NEXT_PUBLIC_MEDUSA_BACKEND_URL
- NEXT_PUBLIC_BASE_URL
- NEXT_PUBLIC_DEFAULT_REGION

这类变量是编译进前端镜像的。只要这些值变了，就必须重新构建前端镜像，不能只改服务器 .env。

## 五、推荐的整体顺序

推荐按下面顺序执行：

1. 申请或准备新域名证书
2. 配置域名 DNS 指向新服务器 IP
3. 新服务器安装 Docker 和 Docker Compose
4. 新服务器开放安全组和系统防火墙端口
5. 上传 deploy 目录到服务器
6. 修改服务器上的 .env、docker-compose、Nginx 配置
7. 上传证书到服务器对应目录
8. 本地构建后端镜像
9. 本地构建前端镜像
10. 推送两个镜像到腾讯云 TCR
11. 在服务器登录腾讯云 TCR
12. 在服务器执行 deploy.sh 发布
13. 检查容器、日志、域名访问、证书、上传文件地址
14. 如果后续程序更新，重新构建镜像并重复发布流程

下面展开每一步。

## 六、第一步：域名与 DNS 解析

### 6.1 建议的域名规划

建议保持和当前架构一致：

- 前台主站：www.example.com
- 前台裸域：example.com
- 后台管理与 API：admin.example.com

### 6.2 DNS 需要配置的记录

在你的域名服务商控制台添加或修改以下记录：

| 主机记录 | 记录类型 | 记录值 | 说明 |
| --- | --- | --- | --- |
| @ | A | 新服务器公网 IP | 裸域 example.com |
| www | A | 新服务器公网 IP | 前台主站 |
| admin | A | 新服务器公网 IP | 后台管理域名 |

如果使用 IPv6，则增加 AAAA 记录。

### 6.3 DNS 生效前不要做的事

在证书申请和正式联调前，不建议先开启 HSTS，也不要立刻把旧站完全下线。应先确认：

- DNS 已解析到新服务器
- 新服务器 80 和 443 可访问
- 证书已上传并可被 Nginx 读取
- Nginx 配置通过测试

## 七、第二步：新服务器初始化

### 7.1 服务器建议配置

建议至少满足：

- Ubuntu Server 24.04 LTS
- 2 核 CPU 以上
- 4GB 内存以上
- 系统盘至少 40GB

如果产品图片较多、数据库较大，建议使用独立数据盘或定期备份 uploads 与 data 目录。

### 7.2 安装 Docker 与 Docker Compose

可以参考现有文档 docs/docker-install-ubuntu2404.md，这里给出最直接命令：

```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
newgrp docker
docker --version
docker compose version
```

### 7.3 创建部署目录

推荐服务器目录：

```bash
mkdir -p /home/opt/ai-cross-stand
```

后续 deploy 目录内容建议放在：

```bash
/home/opt/ai-cross-stand/
```

### 7.4 开放安全组与系统防火墙

云服务器安全组至少开放：

| 端口 | 协议 | 是否必须 | 说明 |
| --- | --- | --- | --- |
| 22 | TCP | 是 | SSH 登录 |
| 80 | TCP | 是 | HTTP，用于跳转和证书校验 |
| 443 | TCP | 是 | HTTPS 正式访问 |

生产环境不建议开放：

- 5432
- 6379
- 9000
- 8000

因为当前生产 docker-compose.yml 中这些服务都走 Docker 内网，不需要公网暴露。

如果服务器系统启用了 ufw，可以执行：

```bash
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status
```

## 八、第三步：上传部署文件到服务器

### 8.1 从本地上传 deploy 目录

在本地仓库根目录执行：

```bash
scp -r deploy/* your-user@your-server-ip:/home/opt/ai-cross-stand/
```

更推荐 rsync：

```bash
rsync -avz --exclude='data/' --exclude='nginx/logs/' --exclude='nginx/ssl/' --exclude='.env' deploy/ your-user@your-server-ip:/home/opt/ai-cross-stand/
```

### 8.2 服务器上进入部署目录

```bash
ssh your-user@your-server-ip
cd /home/opt/ai-cross-stand
```

## 九、第四步：修改服务器上的部署配置

### 9.1 创建 .env

先复制模板：

```bash
cd /home/opt/ai-cross-stand
cp .env.example .env
```

然后编辑：

```bash
nano .env
```

### 9.2 .env 里必须修改的值

如果生产环境改为腾讯云 TCR，建议至少确认这些变量：

```env
IMAGE_TAG=latest

POSTGRES_USER=medusa
POSTGRES_PASSWORD=这里换成强密码
POSTGRES_DB=medusa-my-store

REDIS_PASSWORD=这里换成强密码

JWT_SECRET=这里换成至少32位随机字符串
COOKIE_SECRET=这里换成至少32位随机字符串

STORE_CORS=https://www.example.com
ADMIN_CORS=https://admin.example.com
AUTH_CORS=https://admin.example.com,https://www.example.com

BACKEND_URL=https://admin.example.com/uploads

NEXT_PUBLIC_MEDUSA_BACKEND_URL=https://www.example.com/medusa-api
NEXT_PUBLIC_BASE_URL=https://www.example.com
NEXT_PUBLIC_DEFAULT_REGION=cn
NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY=你的 publishable key
REVALIDATE_SECRET=这里换成随机字符串

TENCENT_REGISTRY_PASSWORD=你的腾讯云镜像仓库密码
```

注意事项：

- BACKEND_URL 必须是后台域名加 /uploads，不是根域名
- NEXT_PUBLIC_MEDUSA_BACKEND_URL 如果前端通过同域 /medusa-api 访问后端，推荐填 https://www.example.com/medusa-api
- AUTH_CORS 建议同时包含前后台域名
- JWT_SECRET、COOKIE_SECRET、REVALIDATE_SECRET 都不要使用模板默认值

### 9.3 把生产 docker-compose.yml 改成腾讯云镜像源

当前 deploy/docker-compose.yml 里的生产镜像默认是 GHCR：

```yaml
admin:
  image: ghcr.io/${GITHUB_REPO}/admin:${IMAGE_TAG:-latest}

storefront:
  image: ghcr.io/${GITHUB_REPO}/storefront:${IMAGE_TAG:-latest}
```

如果你要统一走腾讯云 TCR，建议改成：

```yaml
admin:
  image: hkccr.ccs.tencentyun.com/website-pro/medusa_plus_pro:${IMAGE_TAG:-latest}

storefront:
  image: hkccr.ccs.tencentyun.com/website-pro/medusa_plus_front_pro:${IMAGE_TAG:-latest}
```

如果不改这里，deploy.sh 就不会去腾讯云拉镜像。

### 9.4 修改后台域名 Nginx 配置

编辑：admin.conf

需要替换的内容：

- server_name admin.wolzq.com 改成新后台域名
- ssl_certificate 路径改成新证书目录名称
- ssl_certificate_key 路径改成新证书目录名称

建议最终逻辑保持不变：

- 80 端口只做跳转
- 443 端口反向代理到 admin:9000
- /uploads/ 由 Nginx 直接读取宿主机上传目录

### 9.5 修改前台域名 Nginx 配置

编辑：storefront.conf

需要替换的内容：

- wolzq.com 改成新裸域
- www.wolzq.com 改成新前台主域名
- ssl_certificate 路径改成新证书目录名称
- ssl_certificate_key 路径改成新证书目录名称

保持原有逻辑：

- 裸域跳转到 www
- /medusa-api/ 反向代理到 admin:9000
- / 代理到 storefront:8000

### 9.6 修改拦截 IP 直访配置

编辑：00-block-ip-access.conf

这个文件里引用了一个现有证书来完成 HTTPS 握手。换域名后，这里的证书路径也必须改，否则 Nginx 可能因为证书路径不存在而启动失败。

### 9.7 创建持久化目录

```bash
mkdir -p /home/opt/ai-cross-stand/data/postgres
mkdir -p /home/opt/ai-cross-stand/data/redis
mkdir -p /home/opt/ai-cross-stand/uploads
mkdir -p /home/opt/ai-cross-stand/nginx/logs
mkdir -p /home/opt/ai-cross-stand/nginx/ssl
```

## 十、第五步：上传 SSL 证书

### 10.1 建议的证书目录

当前 Nginx 配置不是使用 api 和 store 这种固定目录，而是直接引用带域名名称的目录，例如：

- /etc/nginx/ssl/admin.wolzq.com_nginx/admin.wolzq.com_bundle.pem
- /etc/nginx/ssl/www.wolzq.com_nginx/www.wolzq.com.key

因此你实际部署时有两种做法：

- 做法 A：沿用现有目录命名风格，把 Nginx 中证书路径改成新域名目录
- 做法 B：统一改为更简洁的固定目录，比如 nginx/ssl/admin 和 nginx/ssl/store，然后同步修改所有 Nginx 配置

为了最少改动，建议做法 A。

### 10.2 上传证书示例

假设新后台域名是 admin.example.com，新前台域名是 www.example.com，可以在服务器创建：

```bash
mkdir -p /home/opt/ai-cross-stand/nginx/ssl/admin.example.com_nginx
mkdir -p /home/opt/ai-cross-stand/nginx/ssl/www.example.com_nginx
```

上传方式示例：

```bash
scp admin.example.com_bundle.pem your-user@your-server-ip:/home/opt/ai-cross-stand/nginx/ssl/admin.example.com_nginx/
scp admin.example.com.key your-user@your-server-ip:/home/opt/ai-cross-stand/nginx/ssl/admin.example.com_nginx/

scp www.example.com_bundle.pem your-user@your-server-ip:/home/opt/ai-cross-stand/nginx/ssl/www.example.com_nginx/
scp www.example.com.key your-user@your-server-ip:/home/opt/ai-cross-stand/nginx/ssl/www.example.com_nginx/
```

### 10.3 上传后检查

```bash
ls -R /home/opt/ai-cross-stand/nginx/ssl
```

## 十一、第六步：本地构建镜像并推送腾讯云 TCR

这一段建议在你的开发机执行。你当前是 Windows，也可以直接在 PowerShell 中执行。

### 11.1 登录腾讯云镜像仓库

```powershell
docker login hkccr.ccs.tencentyun.com --username=<腾讯云账号ID>
```

更安全的方式：

```powershell
echo "<密码>" | docker login hkccr.ccs.tencentyun.com --username=<腾讯云账号ID> --password-stdin
```

### 11.2 构建后端镜像

在仓库根目录执行：

```powershell
docker build -f my-store/Dockerfile -t medusa-admin:latest .
```

打标签并推送：

```powershell
docker tag medusa-admin:latest hkccr.ccs.tencentyun.com/website-pro/medusa_plus_pro:<tag>
docker push hkccr.ccs.tencentyun.com/website-pro/medusa_plus_pro:<tag>
```

建议 tag 使用可追溯格式，例如：

- 20260406
- 20260406-01
- sha-提交短哈希

### 11.3 构建前端镜像

前端镜像必须在构建时写入公开环境变量。推荐在 PowerShell 中显式传参：

```powershell
docker build -f my-store-storefront/Dockerfile `
  --build-arg NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY=pk_xxx `
  --build-arg NEXT_PUBLIC_MEDUSA_BACKEND_URL=https://www.example.com/medusa-api `
  --build-arg NEXT_PUBLIC_BASE_URL=https://www.example.com `
  --build-arg NEXT_PUBLIC_DEFAULT_REGION=cn `
  -t medusa-storefront:latest .
```

然后打标签并推送：

```powershell
docker tag medusa-storefront:latest hkccr.ccs.tencentyun.com/website-pro/medusa_plus_front_pro:<tag>
docker push hkccr.ccs.tencentyun.com/website-pro/medusa_plus_front_pro:<tag>
```

### 11.4 什么时候必须重建前端镜像

只要下面任意一项变化，就必须重新构建前端镜像：

- NEXT_PUBLIC_MEDUSA_BACKEND_URL 变了
- NEXT_PUBLIC_BASE_URL 变了
- NEXT_PUBLIC_MEDUSA_PUBLISHABLE_KEY 变了
- NEXT_PUBLIC_DEFAULT_REGION 变了
- 前端代码变了
- next.config.js 改了

### 11.5 什么时候只需要重启服务，不一定重建镜像

下面这些只要服务器配置正确，通常不需要重新构建前端镜像：

- 仅更换服务器 IP，但域名不变
- 只更新证书
- 只改 Nginx 代理规则
- 只改 PostgreSQL 或 Redis 密码并同步到后端环境

## 十三、第八步：首次发布到新服务器

### 13.1 给脚本执行权限

```bash
cd /home/opt/ai-cross-stand
chmod +x deploy.sh
```

### 13.2 先测试 Nginx 配置

```bash
docker compose --env-file .env -f docker-compose.yml run --rm nginx nginx -t
```

如果这里报错，先不要正式发布，先修正：

- 域名配置
- 证书路径
- 证书文件权限
- 拼写错误

### 13.3 执行生产发布

```bash
cd /home/opt/ai-cross-stand
bash deploy.sh <tag>
```

例如：

```bash
bash deploy.sh 20260406-01
```

deploy.sh 会自动完成：

- 检查 .env 是否存在
- 检查 Docker 与 Docker Compose
- 更新 .env 中的 IMAGE_TAG
- 创建 data、uploads、nginx/logs、证书目录
- 拉取镜像
- 启动 postgres 和 redis
- 等待数据库就绪
- 启动 admin
- 执行数据库迁移
- 启动 storefront
- 启动或重载 nginx

### 13.4 发布完成后检查

```bash
docker compose --env-file .env -f docker-compose.yml ps
docker compose --env-file .env -f docker-compose.yml logs --tail=100 admin
docker compose --env-file .env -f docker-compose.yml logs --tail=100 storefront
docker compose --env-file .env -f docker-compose.yml logs --tail=100 nginx
```

浏览器侧要检查：

- https://www.example.com 是否可打开
- https://admin.example.com/app 是否可打开
- 后台上传图片后，图片 URL 是否是新后台域名
- 前端页面中后台上传图片是否正常显示
- www 与裸域跳转是否符合预期
- HTTP 是否自动跳到 HTTPS

## 十四、第九步：如果修改了程序，如何重新构建、删除旧镜像并重新发布

### 14.1 标准更新流程

如果你改了前后端程序，标准流程是：

1. 本地重新构建镜像
2. 推送新 tag 到腾讯云 TCR
3. 服务器执行 bash deploy.sh 新 tag

推荐命令：

```bash
cd /home/opt/ai-cross-stand
bash deploy.sh 20260406-02
```

这通常已经够用，不一定需要手动删容器。

### 14.2 如果想强制删除旧容器再启动

```bash
cd /home/opt/ai-cross-stand
docker compose --env-file .env -f docker-compose.yml stop admin storefront nginx
docker compose --env-file .env -f docker-compose.yml rm -f admin storefront nginx
bash deploy.sh 20260406-02
```

说明：

- 这不会删除 postgres、redis 数据卷
- uploads、data 目录仍会保留
- 适合容器状态异常或想强制重新创建业务容器时使用

### 14.3 如果想删除旧镜像再重新拉取

先查看镜像：

```bash
docker images | grep medusa_plus
```

删除旧镜像示例：

```bash
docker image rm hkccr.ccs.tencentyun.com/website-pro/medusa_plus_pro:旧tag
docker image rm hkccr.ccs.tencentyun.com/website-pro/medusa_plus_front_pro:旧tag
```

如果镜像还被容器占用，先删容器再删镜像。

### 14.4 如果要彻底重新拉取当前 tag

## 十五、第十步：如果只换服务器和域名，哪些地方一定要改，哪些地方不一定要改

### 15.1 一定要改的

- deploy/.env 里的 CORS、BACKEND_URL、NEXT_PUBLIC_BASE_URL、NEXT_PUBLIC_MEDUSA_BACKEND_URL、Publishable Key、各类密码和密钥
- deploy/nginx/conf.d/admin.conf 里的 server_name 和证书路径

- deploy/nginx/conf.d/00-block-ip-access.conf 的证书路径
- DNS A 记录
- 安全组 22、80、443
- 服务器上的证书文件

### 15.2 通常建议改的

- my-store-storefront/next.config.js 中的 remotePatterns 域名白名单
- deploy/docker-compose.yml 中的镜像源，统一成腾讯云 TCR

### 15.3 不一定要改的

- 后端 Dockerfile
- 前端 Dockerfile 本身
- deploy.sh 脚本主体逻辑
- PostgreSQL 和 Redis 的服务结构

## 二十、结论

这次场景里，真正关键的不是单独改某一个文件，而是同时保证以下四件事一致：

- 域名 DNS 指向了新服务器
- Nginx 使用的是新域名和新证书
- 后端 BACKEND_URL 和前端 NEXT_PUBLIC 变量使用的是新地址
- 生产 docker-compose.yml 拉取的是你实际推送到的镜像仓库

只要这四层保持一致，新服务器、新域名、新证书的迁移就能顺利完成。反过来，只要其中一层仍残留旧域名、旧镜像源或旧证书路径，部署就很容易出现“容器启动了但页面打不开”或“图片地址仍然是旧域名”的问题。
