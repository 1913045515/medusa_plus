## Context

当前首页能力已经具备多配置、草稿和发布机制，但内容模型仍然强绑定 hero、highlights 和 featured_courses 等结构化字段。新需求要求后台能够直接管理外部定制的静态电商首页，并将某一份 HTML 模板发布为 storefront 首页，同时不与商品、课程等业务数据发生运行时联动。

这次改动会同时影响 Medusa 后端数据模型、Admin 编辑界面、Store API 返回结构以及 Next.js 首页渲染逻辑，属于跨模块变更。现有发布流和默认首页兜底能力是可复用资产，应尽量保留。

## Goals / Non-Goals

**Goals:**
- 让管理员在 Medusa Admin 中创建、编辑、预览、保存和发布静态首页模板。
- 将现有首页记录模型扩展为模板模式，支持保存 HTML 与 CSS 内容。
- storefront 首页在存在已发布模板时优先渲染模板，在无模板或模板不可用时回退到默认首页。
- 对模板内容施加可控边界，禁止自定义 JavaScript 执行。

**Non-Goals:**
- 不接入外部 CMS、可视化低代码搭建器或拖拽编辑器。
- 不实现媒体库、组件市场、多页面模板体系或模板版本回滚。
- 不支持模板内容中的动态商品查询、课程查询或第三方脚本执行。

## Decisions

### 1. 复用现有首页记录与发布机制，扩展为模板渲染模式

首页配置仍然保留 `title`、`handle`、`status`、`is_active` 和 `published_at`，避免重建一套新的发布体系。`content` 从固定结构化字段升级为支持模板模式的文档结构，新增 `render_mode` 标识当前记录是 `structured` 还是 `static_html`。

选择原因：
- 可复用现有 repository、service、admin/store 路由和 active published 逻辑。
- 允许平滑兼容旧结构化首页数据，降低迁移成本。

备选方案：
- 新建独立 homepage_template 表：隔离更强，但会复制一套草稿/发布逻辑，增加维护面。

### 2. 模板内容采用 HTML + CSS 双字段，不支持 JavaScript

静态模板记录保存 `html` 与 `css` 两部分，HTML 负责页面结构，CSS 负责模板样式。系统明确移除或拒绝 `script` 标签、内联事件属性和 `javascript:` URL，避免 storefront 运行时执行模板脚本。

选择原因：
- 满足“外部定制好的静态页面可直接放入后台”的核心诉求。
- 将能力边界限制在静态展示层，避免把 storefront 变成不受控执行环境。

备选方案：
- 直接保存完整 HTML 文档：导入更自由，但会和 Next.js 页面壳、全局样式、头尾布局更易冲突。
- 允许自定义 JS：灵活性更高，但存在明显安全和稳定性风险。

### 3. storefront 使用受控模板渲染器，并保留默认首页兜底

首页路由读取 Store API 返回的已发布记录；当 `render_mode=static_html` 时，交给专用模板渲染器输出 HTML 与样式。若接口失败、模板为空或内容被清洗后不可用，则继续渲染当前默认首页内容。

选择原因：
- 与现有“远端内容 + 本地兜底”的策略一致，首页稳定性更高。
- 将模板渲染边界集中在一个 storefront 组件中，便于后续增加过滤和兼容策略。

备选方案：
- 完全依赖后台模板：实现更简单，但首页稳定性完全受后台内容质量影响。

### 4. Admin 编辑页采用源码编辑 + 实时预览，不做拖拽搭建

后台页面由模板列表、基础信息表单、HTML/CSS 编辑区和预览区组成。预览优先基于当前编辑内容即时渲染，发布仍沿用现有草稿/发布动作。

选择原因：
- 与“导入外部静态页面”的工作方式匹配，学习成本低。
- 避免为低代码交互引入额外依赖与复杂状态管理。

备选方案：
- 引入富文本或 page builder：运营体验更强，但明显超出当前范围。

## Risks / Trade-offs

- [风险] 模板 CSS 可能污染 storefront 现有样式 -> 缓解：要求模板根节点使用固定容器类名，并在渲染器中将样式包裹在该命名空间下。
- [风险] 管理员粘贴的 HTML 包含危险标签或属性 -> 缓解：后端保存前和前端渲染前都执行清洗，并拒绝高风险内容。
- [风险] 旧结构化首页记录与新模板模式并存会增加分支复杂度 -> 缓解：保留 `structured` 兼容分支，但 Admin 页面默认只暴露静态模板编辑路径。
- [风险] 无 JavaScript 会限制某些外部模板效果 -> 缓解：明确首期只支持纯静态展示，复杂交互仍需前端代码实现。

## Migration Plan

1. 扩展首页内容类型、模型和 OpenAPI，加入模板模式字段与默认值。
2. 调整 service/repository，支持静态模板记录的创建、保存和发布，同时兼容旧结构化数据读取。
3. 重写 Admin 首页管理页，替换结构化表单为模板编辑器与预览器。
4. 更新 storefront 首页数据访问和渲染逻辑，新增模板渲染器与兜底路径。
5. 验证 Admin API、Store API 和 storefront 首页渲染；如出现回滚需求，可临时强制 storefront 使用现有默认首页。

## Open Questions

- 是否需要在首期就提供“导入整份 HTML 文件”的体验，还是先支持粘贴源码。
- 是否允许保留 `iframe`、`video` 等嵌入标签；若允许，需要补充更细的白名单策略。
