## Context

当前首页静态模板管理已经支持创建、编辑、预览、保存草稿和发布，但模板记录生命周期仍然不完整。运营在实际使用时经常需要基于现有模板快速派生一个新版本，也需要清理废弃草稿，否则模板列表会持续增长并降低可维护性。

这次变更主要影响首页模板的 Admin 管理能力和后台 record lifecycle 逻辑，不改变 storefront 已发布模板的渲染协议。

## Goals / Non-Goals

**Goals:**
- 支持在后台复制现有首页模板记录并生成新的草稿模板。
- 支持在后台删除非活动模板记录。
- 确保复制后的模板自动生成新的 id、标题和 handle，避免冲突。
- 明确删除安全边界，防止误删当前 storefront 正在使用的活动模板。

**Non-Goals:**
- 不实现回收站、恢复删除、版本比较或模板历史追踪。
- 不支持批量删除、批量复制或跨页面模板迁移。
- 不修改 storefront 首页模板渲染逻辑。

## Decisions

### 1. 删除采用软删除语义，并禁止删除活动模板

模板删除会通过 repository 将目标记录标记为已删除，使其不再出现在模板列表中。当前活动模板若仍处于 `is_active=true`，删除操作直接拒绝。

选择原因：
- 现有查询已经按 `deleted_at is null` 过滤，天然适合软删除。
- 可以减少误删带来的破坏性影响，也保留后续扩展恢复能力的空间。

备选方案：
- 物理删除：实现简单，但一旦误删无法恢复，风险过高。

### 2. 复制生成新的草稿记录，并派生标题与 handle

复制动作会读取原模板记录，保留模板内容，生成新 id，并将状态设为 `draft`、`is_active=false`。标题追加“副本”，handle 追加 `-copy`；若已存在冲突，则继续追加递增后缀直到可用。

选择原因：
- 符合运营基于现有模板继续改版的使用习惯。
- 复制后的记录不会影响当前活动版本，安全性更高。

备选方案：
- 复制后覆盖当前记录：更像另存为，但会混淆草稿与已发布模板的关系。

### 3. Admin 页面在列表与编辑区提供显式操作

后台会在模板列表项或编辑区提供“复制模板”和“删除模板”按钮。删除前要求二次确认，复制成功后自动切换到新副本，便于继续编辑。

选择原因：
- 操作入口靠近模板上下文，学习成本低。
- 复制后自动切换可以减少手动查找新记录的步骤。

备选方案：
- 将操作放在独立下拉菜单：界面更紧凑，但不如显式按钮直观。

## Risks / Trade-offs

- [风险] 自动派生 handle 规则不透明，用户可能找不到新副本 -> 缓解：复制成功后提示新标题/handle，并自动选中新副本。
- [风险] 软删除后数据库仍保留历史记录 -> 缓解：首期接受该代价，后续若需要再补清理策略。
- [风险] 用户误删草稿模板 -> 缓解：删除前增加确认，且禁止删除活动模板。

## Migration Plan

1. 扩展 repository/service，增加复制与删除能力。
2. 增加 Admin API 路由，暴露 copy 和 delete 操作。
3. 更新 Admin 模板管理页，接入复制、删除按钮与反馈。
4. 验证复制生成的新草稿和删除后的列表行为是否符合预期。

## Open Questions

- 删除操作是否需要额外校验“已发布但非活动”的模板是否允许删除；当前方案允许删除，只限制活动模板。
