## Context

当前首页由 storefront 代码直接渲染静态 Hero 和商品集合，后台没有任何首页内容维护入口。这个变更同时跨越 `my-store`、`my-store/src/admin` 和 `my-store-storefront`，需要新增数据存储、后台编辑界面、admin/store API，以及前台数据驱动渲染能力。

首页内容编辑的主要使用者是运营或后台管理员，他们需要可视化维护而不是改代码。与此同时，首页作为高频访问页面，前台必须保持渲染稳定，即使后台内容缺失也不能直接白屏。

## Goals / Non-Goals

**Goals:**
- 提供一套首页内容的结构化存储模型，支持 hero、卖点区块、精选课程区块和 CTA 按钮。
- 提供 admin 端可视化表单编辑页面，允许管理员查看、修改并保存首页配置。
- 提供 storefront 只读接口，前台根据后台内容动态渲染首页。
- 在后台内容缺失或不完整时，前台使用本地默认值兜底。

**Non-Goals:**
- 不实现任意区块拖拽排序器或通用 page builder。
- 不实现多版本草稿流、审批流、定时发布和审计历史。
- 不把这一套内容模型复用到其他页面。

## Decisions

### 1. 使用多条结构化首页配置记录，而不是通用 CMS 表模型

首页第一版仍然使用结构化 JSON 内容，但记录模型升级为多条配置。每条记录都具备 `title`、`handle`、`status`、`is_active` 和 `content`，既允许运营维护多个版本，也避免直接走通用 block CMS 的复杂度。

备选方案：
- 通用 block-based CMS：灵活，但会引入复杂 schema、排序、嵌套和渲染协议，超出当前范围。
- 纯代码常量文件：实现最简单，但无法满足后台编辑诉求。

### 2. Admin 维护多条配置，Storefront 仅读取当前活动的发布态

后台允许维护多条首页配置，其中可存在多个草稿，但 storefront 只读取 `is_active=true` 且 `status=published` 的那一条。发布动作会把当前选中配置切为活动版本，并取消其他记录的活动状态。

备选方案：
- 单条配置直接覆盖：更简单，但不利于运营保留多个版本或提前编辑下一版首页。

### 3. Storefront 采用“远端内容 + 本地兜底”的渲染策略

首页请求 store API 获取配置；若接口不可用、返回空值或缺字段，则回退到默认内容，保证首页始终可用。

备选方案：
- 强依赖远端内容：内容中心化更彻底，但首页稳定性依赖后台和数据完整性，风险偏高。

### 4. 内容模型优先面向首页渲染，而不是面向编辑器抽象

数据结构会直接映射到首页需要渲染的区块，包含：
- hero：标题、副标题、描述、背景图、主按钮、次按钮
- highlights：3 个以内卖点卡片
- featured_courses：若干课程卡片，支持标题、描述、图片、跳转

这样 admin 表单和 storefront 组件都能围绕同一份稳定 schema 开发。

## Risks / Trade-offs

- [风险] 结构化 schema 灵活性不足，未来新增首页区块可能需要改模型 → 缓解：在第一版把字段组织为可扩展对象和数组，预留 metadata。
- [风险] 后台录入错误 URL 或空字段会影响展示质量 → 缓解：admin 表单提供必填校验和默认占位提示。
- [风险] storefront 在服务端请求内容接口可能增加首页加载开销 → 缓解：只请求单个配置对象，并保留默认兜底内容避免失败级联。
- [风险] 若后续要支持多国家站点，本次单配置模型不够 → 缓解：当前先实现全站单份配置，后续可扩展 `country_code` 或 `channel` 维度。

## Migration Plan

1. 在 backend 中新增首页内容表或配置存储及对应 service/repository。
2. 新增 admin 读写接口与后台管理页面。
3. 新增 storefront 首页读取接口并切换首页渲染逻辑。
4. 新增扩展迁移，把旧的单条首页记录补齐为 `title / is_active / published_at` 字段，并把已发布记录升级为活动版本。
5. 提供默认初始内容，确保新环境即使没有后台录入数据也能正常显示。
6. 若上线后需要回滚，可将 storefront 首页改回本地静态兜底渲染，并停用新的 admin 入口。
