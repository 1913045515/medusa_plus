## Why

当前商品模型只能表达实体商品，无法在后台明确配置“虚拟资料”和“虚拟课程”两类虚拟商品，也无法在订单详情中稳定展示对应的下载入口或课程跳转地址。这导致虚拟交付依赖隐式逻辑，后台配置不完整，客户购买后也缺少清晰的访问入口。

## What Changes

- 在 admin 产品管理中增加虚拟商品字段：`is_virtual`、`virtual_product_type`，并按类型补充必填关联数据。
- 为虚拟资料类商品增加下载地址配置；为虚拟课程类商品增加课程选择配置，并将课程反向关联中的商品字段改为选填以支持双向绑定。
- 在订单商品快照中存储虚拟商品标记、类型 code，以及资料下载地址或课程相对路径，确保下单后展示数据不受商品后续修改影响。
- 在订单详情中展示虚拟商品信息，并为课程提供跳转入口、为资料提供可下载入口。
- 将虚拟类型统一改为 code 存储，并补齐 admin/storefront 的多语言显示映射。

## Capabilities

### New Capabilities
- `virtual-product-catalog`: 管理员可以在商品中配置虚拟商品属性、类型 code，以及与资料下载地址或课程的必填关联。
- `virtual-order-entitlements`: 订单在完成购买后保存虚拟商品交付快照，并在订单详情中展示可跳转课程地址或可下载资料地址。

### Modified Capabilities

无。

## Non-goals

- 不重做现有课程购买鉴权逻辑，只复用已实现的“未购买提示购买”能力。
- 不引入新的虚拟商品类型范围，当前仅支持 `resource` 与 `course` 两类。
- 不在本次变更中实现文件托管、签名下载或外部 DRM 能力。

## Impact

- Backend: `my-store` 的商品扩展、课程关联、订单落库、store/admin API、OpenAPI 与可能的数据库迁移。
- Admin: 商品编辑表单、订单详情展示、i18n 文案与类型下拉。
- Storefront: 账户订单详情中的虚拟商品展示、课程跳转与资料下载入口。
