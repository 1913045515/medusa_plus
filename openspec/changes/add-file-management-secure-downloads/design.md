## Context

当前虚拟产品"数字资料"的交付依赖直接存储在商品 metadata 中的 `resource_download_url` 字段（外部 URL），并在订单行快照中原样写入。这个设计不支持流量控制、后端鉴权或下载次数审计。

现有的 S3 集成（`add-s3-course-media-storage`）已在 `my-store` 中建立了 AWS SDK 调用和预签名 URL 生成能力，本次文件管理模块可复用该基础设施，无需重新配置 AWS 连接。

变更横跨文件资产模块（后端新增）、虚拟产品配置（后端 + admin 修改）、订单下载（后端 + storefront 修改）和数据库迁移，属于带外部依赖和现有数据改造的跨模块变更。

## Goals / Non-Goals

**Goals:**
- 建立独立的文件资产管理模块，文件只写入私有 S3 桶，admin 具有完整 CRUD 能力。
- 虚拟产品数字资料字段从外部 URL 改为关联文件资产 ID，强化数据引用完整性。
- 受控下载：后端按需生成 30 分钟预签名 URL，记录下载行为，每文件每用户每天限制 3 次。
- 签名过期后，前端展示引导刷新的友好提示，而不是 403 空白页。

**Non-Goals:**
- 不实现浏览器直传 S3 或 CDN 分发。
- 不重购课程媒体链路（已由 add-s3-course-media-storage 处理）。
- 不实现文件公开访问或外部分享。

## Decisions

### D1：文件资产以独立 Medusa 模块（`file-asset`）实现，不绑定商品模块

**选择**：新建 `my-store/src/modules/file-asset/` 模块，独立管理文件元数据（`file_asset` 表）和下载日志（`file_download_log` 表）。商品模块通过 `resource_file_asset_id` 外键引用文件资产。

**理由**：文件资产是跨实体的公共资源，未来可复用于其他业务场景（如电子书、证书等）；独立模块边界清晰，避免商品模块过胀。

**备选方案**：在商品模块内直接存文件元数据 → 耦合强，复用难。

### D2：文件上传由后端代理完成（admin → Medusa → S3），不做浏览器直传

**选择**：admin 的文件上传请求经由 `POST /admin/file-assets/upload` 转发，Medusa 服务端使用已有 AWS SDK 写入私有 S3 桶，返回文件元数据。

**理由**：与现有 S3 媒体上传路径一致（add-s3-course-media-storage 先例）；IAM 密钥不离开服务端；可在服务端统一做文件类型校验和命名规则。

**备选方案**：浏览器直传 + 预签名 PUT → 减少后端带宽，但需要直传协议、回写确认和更复杂的失败处理，不符合最小改动目标。

### D3：下载接口由后端生成 30 分钟预签名 URL 并 302 重定向，不在响应中直接暴露 URL

**选择**：`GET /store/file-assets/{id}/download` 接口在鉴权和下载次数校验通过后，生成 S3 presigned GET URL（30 分钟有效），返回 302 重定向至 presigned URL，而不是在 JSON 响应中返回 URL 字符串。

**理由**：302 模式下 URL 通过 `Location` 头返回，浏览器直接跳转下载，普通用户在历史记录中看不到完整的 S3 签名 URL；且后端在每次调用时记录下载日志，确保计数准确。

**备选方案**：
- 在 JSON 中返回 presigned URL → 前端需二次处理，且 URL 被持久化在响应缓存中，被直接传播的概率更高。
- Nginx 代理流式下载 → 增加带宽和延迟，小文件不值得。

### D4：下载次数计数以 `(customer_id, file_asset_id, date)` 为粒度，每日重置

**选择**：在 `file_download_log` 表中按 `(customer_id, file_asset_id, date(downloaded_at))` 计数，每次下载前查询当日次数，≥ 3 次时返回 HTTP 429 并附带友好消息；date 以 UTC 为基准（可配置）。

**理由**：实现简单，无需定时任务重置计数；每次下载时在同一事务内写日志，保证幂等性。

**备选方案**：Redis 计数器 + TTL → 无需每次查 DB，但引入新的外部依赖，复杂度不匹配当前规模。

### D5：签名过期由前端检测并展示刷新提示，而不是后端静默续期

**选择**：后端在下载 API 响应或 order items 数据中附带 `download_expires_at` 时间戳（当前时间 + 30 分钟，在前端请求后端时生成）；前端在 30 分钟后检测到过期时，展示"下载链接已过期，请刷新页面以重新获取文件"提示，点击"刷新页面"后重新请求订单详情，重新获取新的有效下载入口。

**实际机制**：前端并不直接持有 presigned URL（因为下载是 302 重定向），而是持有"下载按钮可用状态"和时间戳（`download_available_until`）。超过该时间后按钮禁用，提示刷新页面。

**理由**：30 分钟是明确的业务约束，显式刷新语义清晰；避免隐式续期带来的状态复杂度。

### D6：虚拟产品订单快照存储 `file_asset_id`，而不是直接存储 presigned URL

**选择**：订单行快照中的资料字段升级为 `resource_file_asset_id`，指向 `file_asset` 表中的记录 ID。storefront 加载订单时，后端根据 `file_asset_id` 在线生成 presigned URL 并注入响应，前端展示动态生成的下载按钮。

**理由**：presigned URL 有时效性，不适合持久化存储；存 ID 使得文件替换后（管理员删除并上传新文件，订单关联新 ID）也能正确处理；同时下载次数限制也基于 `file_asset_id` 统计。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| admin 删除文件时仍有订单引用该文件 | 删除前检查引用；若有活跃订单引用则拒绝删除，提示管理员先处理 |
| 302 重定向后 presigned URL 出现在浏览器 Network 面板 | 这是技术限制；签名 30 分钟内有效，URL 传播价值低；这不是安全漏洞，属于设计上的已知权衡 |
| 下载日志表在大规模场景下成为性能瓶颈 | 当前规模下无问题；加 `(customer_id, file_asset_id, date)` 复合索引即可，必要时后续引入 Redis |
| 已有虚拟产品 MD 中的 `resource_download_url` 字段失效 | 迁移期间同时支持旧字段（降级展示"配置待更新"），由管理员逐步迁移商品配置 |
| S3 私有桶策略配置不当导致文件可公开访问 | 文档明确要求桶策略禁止公开 Get Object，只允许通过 presigned URL 访问；部署清单中列为必检项 |

## Migration Plan

1. 创建 `file_asset` 和 `file_download_log` 表（数据库迁移）。
2. 部署文件资产模块后端（上传、列表、删除 API）。
3. 部署虚拟产品字段扩展（admin 商品编辑表单改为文件选择器），不删除旧 `resource_download_url` 字段。
4. 运营团队逐步将已有资料类商品的下载地址迁移为文件资产引用。
5. 部署 storefront 订单详情受控下载入口。
6. 确认无旧字段活跃引用后，后续版本中可从数据库迁移移除旧字段（不在本次变更内）。

**回滚**：回滚后端版本后，旧 `resource_download_url` 字段继续有效；新上传的文件在 S3 中已存在，不再被引用但不影响服务运行；可手动清理孤儿文件。

## Open Questions

- 下载次数限制（每日 3 次）是否需要管理员在 admin 中可配置，还是写死在代码中？（本次先写死为 3，后续可提取为全局设置）
- 文件大小上限是否需要与 add-s3-course-media-storage 对齐（2 GB）？（建议先限制为 500 MB for 数字资料，与视频不同场景）
